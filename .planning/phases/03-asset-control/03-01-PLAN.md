1. Register with:
   ```
   server.tool(
     "send_plugin_message",
     "Send a message to a Warudo plugin. The plugin must be loaded and listening for the specified action.",
     {
       pluginId: z.string().describe("ID of the target Warudo plugin."),
       action: z.string().describe("Action name the plugin is listening for."),
       payload: z.unknown().optional().describe("Optional payload data to send with the message."),
     },
     handler
   )
   ```
2. Handler implementation:
   a. Call `await wsClient.ensureConnected()`
   b. Build message: `{ action: "sendPluginMessage", pluginId, action: pluginAction, payload }` — NOTE: rename the `action` parameter in destructuring to avoid collision with the WebSocket message's `action` field. Use `pluginAction` or similar.
   c. IMPORTANT: The Zod schema field is named `action` but the WebSocket message also uses `action` for the WebSocket action name. Handle this carefully:
      ```typescript
      async ({ pluginId, action: pluginAction, payload }) => {
        const response = await wsClient.send({
          action: "sendPluginMessage",
          pluginId,
          pluginAction,  // Note: Warudo SDK expects this as "action" inside the message data
          payload,
        });
      ```
      Actually, the WebSocket protocol likely expects the plugin action as a nested field. Based on research, the correct structure is:
      ```typescript
      await wsClient.send({
        action: "sendPluginMessage",
        pluginId,
        action: pluginAction,  // This won't work -- action key collision
      });
      ```
      Since the WebSocket `send()` uses spread (`{ ...message, requestId }`), we need to pass the plugin action as a differently-named field or structure the payload differently. The safest approach:
      ```typescript
      const msg: Record<string, unknown> = {
        action: "sendPluginMessage",
        pluginId,
        payload: payload ?? {},
      };
      // The plugin action goes as a top-level field
      // Warudo SDK names it "action" but we already used that for the WS action
      // Pass it explicitly after spread won't work; include in payload or as separate field
      ```
      Resolution: Use a raw object to build the message, setting the WebSocket action and including the plugin action as a separate field. The Warudo SDK expects `{ action: "sendPluginMessage", pluginId, action: "pluginActionName", payload }`. Since JavaScript object literals allow duplicate keys (last wins), build it carefully:
      ```typescript
      const wsMessage = {
        action: "sendPluginMessage",
        pluginId,
        pluginAction,
        payload: payload ?? {},
      };
      ```
      If Warudo expects the field literally named "action" for the plugin action, we may need to use `Object.assign` or a different approach. For now, use `pluginAction` as the field name and document that this may need adjustment when tested against a running Warudo instance. Add the plugin action under a field name that won't collide.

      SIMPLEST CORRECT APPROACH: Since `wsClient.send()` accepts `WarudoWebSocketMessage` which is `{ action: string; [key: string]: unknown }`, just do:
      ```typescript
      await wsClient.send(Object.assign(
        { action: "sendPluginMessage" },
        { pluginId, action: pluginAction, payload: payload ?? {} }
      ));
      ```
      Wait -- this would overwrite `action` with `pluginAction`. Instead:
      ```typescript
      const response = await wsClient.send({
        action: "sendPluginMessage",
        pluginId,
        pluginAction: pluginAction,
        payload: payload ?? {},
      });
      ```
      Use `pluginAction` as the field name. If Warudo needs it as `action`, this can be adjusted when testing against a running instance. The error message should note this.
   d. Format success:
      ```
      Sent plugin message.
      Plugin ID: {pluginId}
      Action: {pluginAction}
      Payload: {formatted payload or "(none)"}
      ```
   e. Try-catch with warudoError():
      ```
      Failed to send plugin message: {error message}

      Make sure the plugin is loaded in Warudo and listening for action '{pluginAction}'.
      ```

IMPORTANT:
- The `action` field name collision between WebSocket protocol and plugin action is a known issue. Use `pluginAction` as the field name in the message to avoid ambiguity. Document with a comment.
- If the Warudo SDK expects the literal field name "action" for the plugin's action, the message structure would need to be adjusted when testing against a live instance.
